# Specification: Offline-First Song CRUD

> Status: Proposed

## Goal

Deliver the first local-first song write slice for Lyron Chords by allowing song creation, update, and deletion to happen locally first, while preserving backend-owned authorization, explicit conflict handling, deterministic local routing behavior, and safe sign-out rules.

## Scope

- Allow users to create, update, and delete songs locally without requiring an active internet connection.
- Queue offline actions using `sync_status` for later synchronization with the Supabase backend.
- Keep song write authorization backend-enforced through capability resolution and RLS/policy helpers.
- Ensure all song mutations remain strictly bounded to the currently active `organization_id`.
- Hide locally deleted songs from normal reads immediately while the delete remains pending.
- Support graceful sign-out by warning the user against discarding unsynced local mutations.
- Keep song slugs stable after creation, unique within the active local organization before sync, and reconciled to the canonical server slug after sync.

## Non-Goals
- Real-time collaborative co-editing algorithms (e.g., Operational Transformation, CRDTs).
- Line-by-line or git-style diff merging for conflict resolution (MVP focuses on explicit manual "keep mine or keep theirs" resolution).

## Architecture & Implementation Rules

### 0. Authorization Boundary
Song create, update, and delete synchronization must remain backend-authorized work.

- The Flutter client may expose song-edit affordances only when the signed-in user appears to have song-edit capability, but the client is never the source of truth for authorization.
- The backend must enforce song write access through backend-owned capability resolution and RLS/policy helpers.
- The required backend capability for create, update, delete, and explicit conflict-overwrite actions is `canEditSongs`.
- The sync worker must use a backend-owned mutation contract for song writes. This may be implemented as RLS-protected table writes or dedicated RPCs, but the enforcement point must remain in Supabase/Postgres rather than in Flutter.
- If the backend rejects a mutation because the user no longer has `canEditSongs`, the client must surface a non-retryable authorization error for that pending mutation rather than silently dropping it or treating it as a merge conflict.

### 1. Primary Keys & Identifiers
All new songs created offline will use a locally generated **UUID v4** as the canonical `id`. This eliminates the risk of primary key collisions when pushing batches of new offline records to the Supabase backend.

### 2. Mutation State Management (Drift)
We implement the offline mutation queue using an in-table state approach for the `songs` entity:
- The local `songs` table in Drift utilizes the `sync_status` column.
- Valid status values: `synced` (default, synchronized with backend), `pending_create`, `pending_update`, `pending_delete`, and `conflict`.
- Local deletions operate as soft-deletes (`pending_delete`) so that the entity remains in the database until the backend successfully acknowledges the deletion.
- Rows in `pending_delete` are treated as removed from normal user reads immediately. They must be excluded from the default song list, direct song lookup, slug-based route resolution, and other normal read-model queries while the deletion is pending.
- A `pending_delete` row may still appear in dedicated sync-status or conflict-resolution surfaces so the user can understand, retry, or discard the pending deletion.
- The application will provide an explicitly triggered manual sync mechanism (e.g., a "Sync now" button or pull-to-refresh) on the user interface to ensure users can deterministically push their offline mutations.
- The MVP shipped in this slice relies on the explicit manual sync affordance. Any background reconnect-driven mutation sync remains a follow-up slice rather than part of this branch's shipped behavior.
- Mutations carrying authorization or dependency errors remain visible in sync-status surfaces. In this MVP, the user may still explicitly re-run sync from the manual sync affordance, but there is no background worker silently retrying them.

### 3. Update And Delete Semantics
- A local create writes a new row with a locally generated UUID, a locally generated base slug, and `sync_status = 'pending_create'`.
- A local update keeps the canonical `id` and `slug`, records the current `base_version`, and sets `sync_status = 'pending_update'`.
- A local delete marks the row `pending_delete` instead of physically removing it until the backend acknowledges the deletion.

#### Related Entity Rules
- Deleting a song is allowed only when no `session_items` in the same organization still reference that song.
- The client should block creation of a `pending_delete` mutation when the local read model already shows a referencing `session_item`.
- The backend must re-check the same dependency rule at sync time so a stale local cache cannot delete a song that has since been referenced elsewhere.
- If the backend rejects a delete because the song is still referenced by `session_items`, the client must surface a non-retryable dependency error rather than reclassifying that failure as a version conflict.
- Successful song deletion cascades server-side to song-owned `attachments`. The local cache must remove those attachment references after the deletion sync succeeds.

### 4. Sync Conflicts & Resolution
Optimistic Concurrency Control (OCC) is enforced to prevent arbitrary overwriting of existing data:
- **Applicability:** OCC checks only apply to `update` and `delete` operations. A `pending_create` mutation generates a novel local UUID, meaning it physically cannot experience an OCC version collision on the server.
- The local app tracks the `base_version` (the server version the offline edit or deletion originally started from).
- Upon syncing an update or delete, the Supabase backend compares the proposed `base_version` against the current server `version`. If the `base_version` is outdated, the backend rejects the write.
- Specifically, if a user attempts to **delete** a song (`pending_delete`) that has been updated concurrently on the server, the deletion will be rejected as a conflict to avoid accidental destruction of new information.
- Upon rejection, the client updates the local row's `sync_status` to `conflict`.
- **User Experience:** The user is presented with a manual conflict resolution UI, effectively asking: *"This song was modified by someone else recently. Keep your offline version/deletion (overwriting the server) or discard your changes (pulling the latest server data)?"*
- While a row remains in `conflict`, ordinary edit and delete flows must not silently continue it as a fresh pending mutation. The user must explicitly choose keep/discard before ordinary editing or deletion can resume.
- **Keep Mine / Keep Deletion Path:** Choosing "keep mine" must trigger a second explicit backend mutation path rather than silently retrying the stale write. That overwrite path must still require `canEditSongs`, must record an auditable version bump, and must return the canonical post-write server representation.
- **Discard Mine Path:** Choosing "discard mine" must replace the local row with the latest server state, clear the local conflict, and restore `sync_status = 'synced'`.
- If the conflict concerns a pending deletion and the user chooses to keep the deletion, the explicit overwrite path must delete the current server version only after re-checking the `session_items` dependency rule.
- If keep/discard fails because of authorization, dependency, connectivity, or unknown backend issues, the client must persist that failure on the local conflict row and surface a recoverable UI error instead of crashing or silently clearing the conflict.

### 5. Slug Generation & Uniqueness
The `slug` is used for public bounded URLs and must be unique within an `organization_id`.
- **Immutability:** To ensure link stability, the `slug` is derived during the initial `pending_create` phase and is considered immutable thereafter. Modifying an existing song's title via an offline update will **not** alter its URL slug.
- **Client-Side Generation:** The local client automatically generates a base slug from the provided song title upon creation (e.g., `amazing-grace`).
- **Local Collision Handling:** The local client must keep slugs unique within the active local organization view before sync succeeds. If another local or cached song in the same organization already uses the base slug, the client must allocate the next available numeric suffix locally (for example `amazing-grace-2`, `amazing-grace-3`) before persisting the new `pending_create` row.
- **Server-Side Mitigation:** When the creation mutation syncs, the backend handles any latent slug collisions gracefully. Through a Supabase trigger, RPC, or application-level SQL retry logic, uniqueness conflicts will safely auto-increment the slug postfix (`amazing-grace-2`) on the server rather than failing the synchronization batch.
- **Client Reconciliation:** When the backend returns a canonical slug that differs from the locally generated slug, the client must update the local song row to the server-returned slug and treat that returned slug as authoritative for future route generation and lookups.

### 6. Sign-Out Safety
Explicit sign-out clears the local authenticated device cache to preserve tenant boundaries on shared devices.
- Before executing a sign-out, the app checks for any local entity where `sync_status != 'synced'`.
- If unsynced data exists, a blocking warning intercepts the navigation: *"You have unsynced modifications. Signing out will permanently discard these changes. Connect to the internet to sync, or confirm that you explicitly want to discard your work."*

## Testing Requirements
- Unit tests must cover local mutation state transitions, `pending_delete` read filtering, OCC decision logic, slug reconciliation, and dependency-check decisions for song deletion.
- Widget tests must cover manual sync affordances, conflict surfaces, dependency-blocked delete messaging, and the sign-out warning when unsynced song mutations exist.
- Integration tests must cover offline song create, update, delete, explicit sync, conflict resolution, and sign-out discard behavior against the local-first repository boundary.
- Backend verification must cover `canEditSongs` enforcement, scoped slug uniqueness, delete rejection while `session_items` still reference the song, and successful attachment cleanup after accepted song deletion.
